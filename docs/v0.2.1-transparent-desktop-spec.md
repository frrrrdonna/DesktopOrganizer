# v0.2.1 Transparent Desktop Mapping Spec

## 1. Version Goal

This version should move Desktop Organizer from a visible desktop-host shell toward a Fences-like transparent desktop mapping layer.

The target user experience is:

- keep the original Windows wallpaper visible;
- hide scattered desktop icons;
- render a small number of floating baskets on top of the desktop;
- treat desktop files as mapped content first, not physically moved content;
- restore the desktop back to normal when the app exits.

## 2. Assumptions

- Target platform is Windows 10/11.
- This version should not replace Explorer as the shell.
- This version should not physically reorganize user desktop files yet.
- This version should prefer safe visual mapping over risky file movement.

## 3. Core Product Principles

1. Preserve wallpaper.
2. Hide and restore desktop icons safely.
3. Treat baskets as transparent floating overlays, not as a replacement desktop background.
4. Prefer logical mapping of desktop items before any physical relocation feature.
5. Make exit recovery reliable before adding more visual complexity.

## 4. Features That Should Be Included In This Version

These features are appropriate to build together in the same version.

### A. Transparent Desktop Overlay

Required behavior:

- the host surface must appear visually transparent or near-transparent;
- the original wallpaper must remain visible behind the baskets;
- there should be no large opaque background panel covering the desktop;
- the host should feel like a desktop layer, not a normal app page.

Development notes:

- remove the solid desktop-colored host background;
- use transparent window styles where WPF allows it;
- keep only basket cards, shadows, outlines, and lightweight chrome visible.

### B. Desktop Icon Mapping

Required behavior:

- scan current desktop content from user and common desktop locations;
- create basket content from scanned files, folders, and shortcuts;
- do not physically move desktop files in this version;
- basket items should open the original files/shortcuts.

Development notes:

- current `DesktopScanner` and `DesktopClassifier` are the base;
- classification quality should improve, but file movement should remain disabled;
- keep path references stable and reversible.

### C. Startup Takeover And Exit Recovery

Required behavior:

- on startup, hide the original desktop icon layer;
- on exit, restore desktop icon visibility;
- startup should attempt desktop-layer attachment;
- if attachment fails, the app should still remain usable without destroying desktop state.

Development notes:

- keep restore logic conservative;
- do not leave the desktop hidden after abnormal close if avoidable;
- add a desktop-state snapshot model before expanding behavior.

### D. Floating Basket UI

Required behavior:

- baskets must look like floating panels, not blocks of a replacement desktop;
- baskets must support collapse;
- baskets must support edge docking;
- baskets must keep readable content over arbitrary wallpapers.

Development notes:

- use semi-transparent backgrounds;
- add rounded corners, subtle shadow, and a clear title bar;
- prioritize left/right docking before more advanced freeform movement.

### E. Settings Window

Required behavior:

- toggle desktop takeover on startup;
- toggle icon hiding;
- toggle auto-generated baskets;
- toggle restore-on-exit;
- expose a future-facing option placeholder for physical file organization, but keep it disabled or clearly marked unavailable.

Development notes:

- do not add large settings infrastructure yet;
- keep the current settings window and expand only the fields needed by this version.

### F. Desktop Snapshot And Restore Safety

Required behavior:

- capture whether desktop icons were visible before takeover;
- capture scanned desktop item state for this session;
- allow safe restoration of icon visibility on exit;
- prepare a model for future “restore original layout” behavior.

Development notes:

- this version needs snapshot metadata, not full physical file rollback;
- focus on state capture, not desktop file migration.

## 5. Features That Should Not Be Included In This Version

Do not implement these yet:

- physical movement of desktop files into managed folders;
- full shell replacement;
- multi-monitor custom placement rules;
- cloud sync;
- AI auto-sorting;
- installer/platform packaging changes unrelated to the desktop overlay behavior;
- complex animation systems that delay functional progress.

## 6. Implementation Order

The work should be done in this order.

1. Desktop snapshot and restore safety
2. Transparent desktop host styling
3. Stable desktop-layer attach and icon hide/restore behavior
4. Floating basket visual redesign
5. Edge docking and collapse refinement
6. Settings updates for takeover behavior
7. Manual regression and publish verification

## 9. Acceptance Criteria

This version is complete only if all conditions below are met.

1. The wallpaper remains visible while the app is running.
2. Desktop icons are hidden on startup and restored on exit.
3. Baskets are rendered as floating overlays, not as a solid background host.
4. Desktop items are scanned and mapped into baskets without moving real files.
5. Basket items still open their original targets.
6. Baskets support collapse and visible edge docking behavior.
7. Settings expose the main desktop takeover toggles.
8. Build, test, and startup smoke checks all pass.

## 10. Explicit Non-Goal For This Version

This version is not the place to physically clear the desktop by moving user files around.

The correct interpretation of “desktop becomes clean” in this version is:

- hide the original icon layer;
- map the original desktop items into baskets;
- preserve wallpaper;
- restore the desktop when the app exits.
